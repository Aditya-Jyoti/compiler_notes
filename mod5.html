Code Optimization

Compiler Design · Unit V

1 Introduction to Code Optimization

🧠 Intuition First
Think of code optimization like a GPS that finds not just a route, but the fastest route. The compiler takes your intermediate code and tries to make it run faster and use less memory — without changing what it does.

The code optimization phase improves the intermediate code so that the resulting machine code is faster and smaller. It does not change the meaning of the program — it only changes how it is computed.

Goal: Less time + Less space, same output.
Better intermediate code → Faster machine code

Criteria for Code-Improving Transformations

#CriterionWhat it means
1Meaning-preservingMust not change the output of the program
2Measurably fasterMust provide a demonstrable speed improvement
3Worth the effortCompilation time overhead must be justified

Organization of the Code Optimizer

Types of Optimization

🏠 Local Optimization

Applied within a single basic block. Fastest to compute, done first.

Example: Replacing x=x+0 with nothing.

🌐 Global Optimization

Applied across multiple basic blocks. Requires data flow analysis.

Example: Moving a loop-invariant computation outside a loop.

🔁 1. Common Subexpression Elimination (CSE)
Definition: If an expression was previously computed and the values of its variables have not changed, reuse the result instead of recomputing it.
🧠 Intuition
If you already calculated 4 * i earlier in the same block, why calculate it again? Store it in a temp variable and reuse it!

Two conditions for CSE:

  1. The expression was previously computed.
  2. The values of variables have not changed since that computation.
📝 Example 1 — Simple CSE
a = b + c d = b + c ← same as above! e = a * d
⇒
t = b + c a = t d = t ← reuse t e = a * d

b+c is computed only once. d gets the same value via t.

📝 Example 2 — Classic from Textbook
a = b + c b = a - d c = b + c ← b changed! not CSE d = a - d ← CSE with line 2
⇒
a = b + c b = a - d c = b + c d = b ← reuse b (which = a-d)

⚠️ b+c is NOT a common subexpression because b changed on line 2. But a-d IS a common subexpression!

📝 Example 3 — Loop context (from notes)

In block B5 of a flow graph, both t7 := 4*i and t10 := 4*j appear redundantly. After CSE:

t6 := 4*i x := a[t6] t7 := 4*i ← same as t6! t8 := 4*j t9 := a[t8] a[t7] := t9 t10 := 4*j ← same as t8! a[t10] := x
⇒
t6 := 4*i x := a[t6] ← t7 removed t8 := 4*j t9 := a[t8] a[t6] := t9 ← use t6 ← t10 removed a[t8] := x ← use t8
📋 2. Copy Propagation
Definition: After a copy statement f := g, replace subsequent uses of f with g (wherever f is not redefined and g is still available).
🧠 Intuition
If you set f = g, then whenever you see f later, you can just say g instead — one fewer variable to track!
📝 Example
f = g ... y = f * 2 ← f is just g
⇒
f = g ... y = g * 2 ← propagated g
📝 Example — After CSE introduces copies

CSE often introduces copy statements. Consider:

x = a y = x * b z = x * c
⇒
x = a y = a * b ← x replaced by a z = a * c ← x replaced by a
⚠️ Important constraint: Copy propagation is valid only if neither f nor g is redefined between the copy statement and the use point.
💀 3. Dead Code Elimination
Live variable: A variable is live at a point if its value may be used later.
Dead variable: A variable whose value is never used after being assigned.
🧠 Intuition
If you do a calculation but never use the result, that computation is wasted effort — remove it entirely!
📝 Example 1
x = a + b ← x never used again! y = c * d return y
⇒
← x=a+b deleted! y = c * d return y
📝 Example 2 — From textbook

In block B5 after copy propagation, x := t3 is dead because x is overwritten before being used.

x := t3 ← dead! a[t2] := t5 a[t4] := x goto B2
⇒
← removed a[t2] := t5 a[t4] := x goto B2
🔢 4. Constant Folding
Definition: Evaluate constant expressions at compile time and replace them with their computed values.
🧠 Intuition
Why compute 3.14159 * 2 every time the program runs? The compiler can calculate it once: 6.28318.
📝 Examples
PI = 22/7 area = PI * r * r x = 2 * 3 + 1
⇒
PI = 3.14 ← folded! area = 3.14 * r * r x = 7 ← folded!

Also applies to boolean/relational constants: if 0 != 1 → always true → branch simplified.

3 Basic Blocks & Flow Graphs

Basic Block: A maximal sequence of consecutive statements where control enters at the beginning and exits at the end — no branches in or out in the middle.
🧠 Intuition
A basic block is like a straight road — you enter at one end and drive through without any turns or exits until you reach the end. A flow graph then connects these roads with junctions (conditionals, gotos).

Algorithm to Identify Basic Blocks

Step 1: Find all leaders (first statements of basic blocks):
  ① The first statement of the program is a leader.
  ② Any statement that is the target of a conditional or unconditional goto is a leader.
  ③ Any statement immediately following a goto or conditional goto is a leader.

Step 2: Each leader starts a block that runs until (but not including) the next leader.
📝 Example — Find Basic Blocks (Exam-style)

Given three-address code:

10 i = 0 20 a = n - 3 30 if i < a goto 50 ← goto target: 50 is leader; next stmt (40) is leader 40 goto 170 ← next stmt would be leader 50 b = i - 4 60 x = a + 2 70 p = x 80 c = p + b 90 d = m[c] 100 e = d - 2 110 f = i - 4 120 g = p + f 130 m[g] = e 140 i = i + 1 150 a = n - 3 160 if i < a goto 50 ← goto target: 50 already leader 170 ----

Leaders: 10 (first), 40 (after conditional), 50 (target), 170 (target of goto 170)

Flow Graph

A flow graph connects basic blocks with directed edges:

  • Block A → Block B if execution can flow from the end of A to the start of B
  • Edges come from sequential flow or from goto/branch targets

Type 1

Structure Preserving

Algebraic Identities

Use mathematical identities to simplify:

x + 0 = x x * 1 = x x ** 2 = x * x (cheaper!) x / 2 = x * 0.5

4 DAG Representation of Basic Blocks

DAG (Directed Acyclic Graph): A graph representation of a basic block where nodes represent values/operations and edges represent dependencies. DAGs detect common subexpressions automatically!
🧠 Intuition
A DAG is like a recipe dependency chart. If two dishes need the same ingredient preparation, you draw one node for that preparation and point both dishes to it — no duplication!

DAG Construction Rules

For each statement x := y op z:
  1. Find or create nodes for y and z
  2. Check if there's already a node for y op z — if yes, that's the CSE node!
  3. If no, create a new internal node labeled op with children y, z
  4. Label this node with x (remove x from any previous node)
📝 Example — Build DAG for basic block

Basic block:

a := b + c b := a - d c := b + c d := a - d

Step-by-step DAG construction:

StatementActionCSE?
a := b + cCreate node (+) with children b, c. Label it aNo
b := a - dCreate node (-) with children a, d. Label it bNo
c := b + cCreate node (+) with children b(new), c. Label it cNo — b changed!
d := a - dNode (a-d) already exists! Label it d too✅ YES — CSE found!

The node labeled b, d shows that both b and d hold the value of a - d. This is the common subexpression detected by the DAG!

Benefits of DAG

  • Automatically detects common subexpressions
  • Enables dead code elimination (nodes with no outgoing references)
  • Determines which variables are computed in the block
  • Allows reordering of independent computations

5 Peephole Optimization

Peephole Optimization: A local optimization technique that examines a small "window" (peephole) of target code instructions and replaces inefficient instruction sequences with better ones.
🧠 Intuition
Imagine proofreading a document through a magnifying glass — you see only a few lines at a time, but you spot and fix local redundancies like "the the" or "is is". Peephole optimization does the same for machine code!
🔑 Mnemonic: "R-FAS-UR"
Redundant Instruction Elimination · Flow-of-control Optimization · Algebraic Simplification · Strength Reduction · Use of Machine Idioms · Remove Unreachable Code
1️⃣ Redundant Instruction Elimination
MOV R0, a ← store a into R0 MOV a, R0 ← store R0 back into a (REDUNDANT!)
⇒
MOV R0, a ← keep only this second instruction deleted!

The second instruction reloads what was just stored. Since R0 already has the value of a, the second MOV is useless.

2️⃣ Flow-of-Control Optimization
goto L1 ... L1: goto L2 ← chain of jumps
⇒
goto L2 ← skip to final target ... L1: goto L2

Eliminate jump chains — jump directly to the final destination.

3️⃣ Algebraic Simplification
x = x + 0 ← useless y = y * 1 ← useless z = z ** 1 ← useless
⇒
← all three deleted!
4️⃣ Use of Machine Idioms

Modern CPUs have special instructions for common patterns like increment/decrement. Use them!

a = a + 1 ← generic add
⇒
INC a ← hardware increment!

Auto-increment/decrement addressing modes also help with stack push/pop operations.

5️⃣ Unreachable Code Elimination
#define DEBUG 0 ... if(DEBUG) print("info") ← never runs!
⇒
← entire debug print removed!

After constant propagation substitutes DEBUG=0, the condition becomes if(0) which is always false — the block is unreachable and deleted.

Original: if debug=1 goto L1 goto L2 L1: print debugging info L2: ... After CP: if 0≠1 goto L2 print debugging info ← unreachable! L2: ... Final: goto L2 ← all simplified
6️⃣ Reduction in Strength (Peephole)
x = y ** 2 ← function call!
⇒
x = y * y ← single multiply!

Replace expensive operations (exponentiation, division) with cheaper equivalents (multiplication).

6 Loops in Flow Graphs

🧠 Why loops matter most?
A loop body that runs 10,000 times — even a tiny saving per iteration (like removing one multiplication) saves 10,000 operations! This is why loop optimization gives the biggest performance gains.

Dominators

Dominator: Node d dominates node n (written d dom n) if every path from the entry node to n passes through d.
📝 Example — Dominator Tree

For a flow graph with nodes 1–10, the dominator relationships form a tree. Node 1 dominates all (it's the entry). Node 7 dominates 7, 8, 9, 10 (only path to those goes through 7).

Natural Loops

A natural loop of a back edge n → d is the set of all nodes that can reach n without going through d, plus d itself. d is the loop header.
Back Edge: An edge a → b where b dominates a. Back edges identify loops!

Two essential properties of a loop:
1. Single entry point called the header (dominates all nodes in the loop)
2. At least one way to iterate (path back to header)
🚀 Loop Optimization 1: Code Motion
Code Motion: Move a loop-invariant computation (one that gives the same result regardless of loop iterations) to the preheader — before the loop.
🧠 Intuition
If you're computing the same thing every iteration (like limit - 2), compute it once before the loop and store it. Don't repeat unchanged work!
📝 Example
while (i <= limit - 2) { // loop body i++; }
⇒
t = limit - 2; ← moved out! while (i <= t) { // loop body i++; }

limit - 2 is computed once instead of every iteration. If the loop runs 1000 times, we save 999 subtractions!

Condition for Code Motion: The expression must be loop-invariant — none of its variables change value inside the loop.
📉 Loop Optimization 2: Induction Variables & Reduction in Strength
Induction Variable: A variable x that changes by a constant amount each loop iteration (e.g., i = i + 1). Any variable linearly related to an induction variable is also an induction variable.
Reduction in Strength: Replace an expensive operation (like multiplication) with a cheaper one (like addition) by exploiting the induction variable relationship.
🧠 Intuition
If i increases by 1 each time, then 4*i increases by 4 each time. Instead of doing t = 4*i (a multiply) every iteration, maintain t directly by adding 4: t = t + 4. Additions are faster than multiplications!
📝 Example — Strength Reduction on 4*j
// Block B3 (loop) j := j - 1 t4 := 4 * j ← multiply every iter! t5 := a[t4] if t5 > v goto B3
⇒
// After strength reduction j := j - 1 t4 := t4 - 4 ← just subtract! t5 := a[t4] if t5 > v goto B3

Since j decreases by 1 each time, 4*j decreases by 4 each time. Replace multiply with subtract. t4 is initialized before the loop as t4 := 4*j (done only once).

📝 Example — Induction Variable Elimination

After strength reduction, if i and j are only used to control the loop test (comparing with each other via t2 >= t4), the original variables i and j become dead — eliminate them!

// B4 test if i >= j goto B6 ← uses i and j
⇒
if t2 >= t4 goto B6 ← use t2,t4 instead i and j are now dead!
TechniqueWhat changesSaving
Code MotionLoop-invariant expr moved outN × (cost of expr)
Strength ReductionMultiply → Add per iterationN × (multiply - add cost)
Induction Var Elim.Remove redundant variablesN × (var update cost)

7 Introduction to Data Flow Analysis

Data Flow Analysis: A framework for collecting information about the flow of data values through a program. Used to detect reaching definitions, live variables, available expressions, etc.
🧠 Intuition
Think of data flow analysis like tracking water through pipes. You want to know: at any point in the program, which "definitions" (assignments) of a variable can "flow" to reach that point without being "blocked" (overwritten)?

Key Concepts

📌 Points

A point is a position between two adjacent statements, or before the first / after the last statement in a basic block.

🛤️ Paths

A path from P1 to Pn is a sequence of consecutive points through the flow graph.

Reaching Definitions

A definition d of variable x reaches a point p if there exists a path from d to p along which x is not redefined.

Data Flow Equation

The fundamental data flow equation:

out[S] = gen[S] ∪ (in[S] − kill[S])
TermMeaning
out[S]Definitions leaving statement S
gen[S]Definitions generated by S itself
in[S]Definitions arriving at S from predecessors
kill[S]Definitions of the same variable killed by S
📝 Example — gen, kill, in, out
Statement: d1: a = b + c gen[d1] = {d1} // this statement defines 'a' kill[d1] = Da - {d1} // all OTHER definitions of 'a' are killed // where Da = all definitions of 'a' in the program out[d1] = {d1} ∪ (in[d1] - kill[d1])

Use-Definition (UD) Chains

A UD-chain for a use of variable x is the list of all definitions of x that can reach that use.

UD-chains are used to:

  • Detect loop-invariant computations (all reaching defs are outside the loop)
  • Perform copy propagation
  • Eliminate induction variables

Bit Vector Representation

🧠 Intuition
Number all definitions in the program (d1, d2, d3 ...). Use a bit vector where bit i = 1 means definition di is in the set. Set union = bitwise OR. Set intersection = bitwise AND. Very efficient!
📝 Example
4 definitions: d1, d2, d3, d4 gen[B1] = {d1, d3} → bit vector: 1 0 1 0 kill[B1]= {d2, d4} → bit vector: 0 1 0 1 Union (OR): 1 0 1 0 | 0 1 0 1 = 1 1 1 1 Intersect(AND): 1 0 1 0 & 0 1 0 1 = 0 0 0 0

8 Solved Questions & Practice Q&A

📋 Q1 — 2 Marks · Definition

What is code optimization? What are the criteria for code-improving transformations?

A
Code Optimization is the process of improving intermediate code so that the resulting machine code runs faster and/or uses less memory, without changing the program's output.

Criteria:
1. Meaning-preserving — the transformation must not change what the program computes.
2. Measurably faster — it must demonstrably speed up execution.
3. Worth the effort — the compilation overhead must be justified.
📋 Q2 — 2 Marks · Definition

Define: (a) Basic Block (b) Dead Code (c) Induction Variable (d) Loop Invariant Computation

A

(a) Basic Block: A maximal sequence of consecutive three-address instructions where control enters at the beginning and leaves at the end without halt or possibility of branching.

(b) Dead Code: A statement that computes a value that is never subsequently used. The variable assigned is "dead" — its value will never be read.

(c) Induction Variable: A variable that changes by a constant amount (is incremented or decremented by a fixed value) every time a loop iterates. Example: i = i + 1 makes i an induction variable.

(d) Loop Invariant Computation: An expression inside a loop whose value does not change across loop iterations — it produces the same result every time, regardless of how many times the loop runs. Example: limit - 2 where limit is not modified in the loop.

📋 Q12 — 2 Marks · Concept

What are dominators in a flow graph? Why are they important?

A

Node d dominates node n if every possible path from the entry node of the flow graph to n must pass through d. Written as d dom n.

Importance:

  • Used to identify natural loops (back edges: n→d where d dominates n)
  • The loop header dominates all nodes in the loop
  • Used in code motion to verify a computation dominates all loop exits (so moving it is safe)
  • Foundation for reducible flow graph analysis

Key fact: Every node dominates itself.

📋 Q13 — 5 Marks · Copy Propagation + Dead Code

Apply Copy Propagation and Dead Code Elimination to:
x = a  |  y = x * b  |  z = x * c  |  w = y + z  |  x = b + c  |  return w

1
Original code:
1: x = a 2: y = x * b ← x is a, so y = a * b 3: z = x * c ← x is a, so z = a * c 4: w = y + z 5: x = b + c ← x redefined (old x = a becomes dead here if not used after) 6: return w ← w uses y and z (which used old x = a)
2
Apply Copy Propagation: Statement 1 is a copy x = a. After this, x can be replaced by a at lines 2 and 3 (before x is redefined at line 5).
3
After Copy Propagation:
1: x = a 2: y = a * b ← x replaced by a 3: z = a * c ← x replaced by a 4: w = y + z 5: x = b + c ← x computed but... 6: return w ← w only uses y, z — not x!
4
Apply Dead Code Elimination: After line 5, x is never used again (the function returns w). So x = b + c at line 5 is dead code. Also, the original x = a at line 1 may now also be dead if x is not used elsewhere.
