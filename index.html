Module 6: Code Generation

Code generation is the final phase of compilation. The compiler takes the optimized intermediate representation (IR) and translates it into target machine code. Think of it like a translator converting English text (IR) into a local dialect (machine code) as efficiently as possible.

Compiler pipeline — Code Generator is the final translation step

1. Issues in the Design of a Code Generator

A code generator must solve several challenging problems simultaneously. The overarching constraint is correctness first — wrong-but-fast code is useless.

🧠 Mnemonic — Remember the 5 Issues
I T R E O

Input (IR format) · Target program (architecture) · Register allocation · Evaluation order · Optimal code generation

1. Input to Code Generator

The code generator receives the IR (Intermediate Representation) + the symbol table. Common IRs:

  • Three-address code: x = y op z
  • DAG (Directed Acyclic Graph)
  • Bytecode / stack machine code

By this stage, all type-checking, semantic errors, and type conversions are done.

2. Target Program

The output can be:

  • Absolute machine code — directly executable, fixed memory location
  • Relocatable object code — linked later, compiled separately
  • Assembly code — human-readable, then assembled

3. Register Allocation

Registers are the fastest but fewest resources. Deciding which values stay in registers vs memory is NP-complete in the general case.

4. Evaluation Order

The order of computing operations affects how many temporary registers are needed. Bad ordering → more spills to memory.

2. Target Machine Architectures

Understanding the target architecture is crucial because the same program generates very different code on different machines.

RISC — Reduced Instruction Set Computer

  • Many registers (typically 32+)
  • All instructions are 3-address: ADD R1, R2, R3 means R1 = R2 + R3
  • Simple addressing modes
  • Fixed-length instructions (e.g., 32 bits)
  • Examples: ARM, MIPS, RISC-V
# RISC: a = b + c
LD  R1, b       ; Load b into R1
LD  R2, c       ; Load c into R2
ADD R3, R1, R2  ; R3 = R1 + R2
ST  a, R3       ; Store result to a

CISC — Complex Instruction Set Computer

  • Few registers, but instructions can access memory directly
  • 2-address instructions: destination is also a source
  • Variable-length instructions
  • Examples: x86, x86-64
; CISC (x86): a = b + c
MOV EAX, b    ; EAX = b
ADD EAX, c    ; EAX = EAX + c  (memory operand allowed!)
MOV a, EAX   ; a = EAX

Stack-Based Machine

  • Operations work on a stack — push operands, operation pops and pushes result
  • No explicit register names needed
  • Java Virtual Machine (JVM) is a famous example
// Stack: a = b + c
PUSH b   ; stack: [b]
PUSH c   ; stack: [b, c]
IADD     ; pops b,c, pushes (b+c): stack: [b+c]
POP a    ; a = b+c

Instruction Selection Example

Given three-address code: a = b + c, the code generator must pick the right instructions. If the machine has an INC instruction and the code is a = a + 1, using INC a is far better than LOAD R, a / ADD R, #1 / STORE a, R.

💡 Intuition — Instruction Selection
Think of it like a restaurant order: the IR says "one egg dish." But the target machine (restaurant) might have a specialized "sunny-side-up" shortcut that's faster. The code generator looks for these shortcuts (machine idioms).

3. Next-Use Information

Before generating code, the compiler needs to know: "After this instruction, will this variable be used again?" This is called Next-Use Information (also called liveness).

💡 Intuition — Why Next-Use Matters
Imagine you're writing on a whiteboard (= registers). If you won't need what's written again, you can erase it (free the register). If you will need it, keep it! Next-use info tells the code generator when it's safe to free a register.

Algorithm: Computing Next-Use

Scan each basic block backwards (from last instruction to first):

  1. Initially, all variables are "dead" (no next-use).
  2. When you see an assignment x = y op z:
    • Record the current next-use info for x, y, z
    • Mark x as dead (it was just defined)
    • Mark y, z as live (they were just used)

Example

Statement   | Variable a | Variable b | Variable c
-----------+------------+------------+------------
(1) a = b+c | dead (def) | use at 1   | use at 1
(2) d = a-b | live@2     | dead (def) | live@1
(3) e = b+c | dead       | use@2,3    | use@1,3

Scanning backwards from (3) to (1) tells us:
After stmt 1: a is live (used in stmt 2), b is live (used in stmt 3)

This lets the register allocator know: if a won't be used after statement 3, we don't need to keep it in a register — we can spill it to memory or just drop it.

4. Register Allocation and Assignment

Registers are the fastest but most limited resource. Register allocation is one of the hardest problems in compilation.

Two Sub-Problems

Register Allocation — Decide which variables get to live in registers at each program point.

Register Assignment — Decide which specific register (R1, R2, R3...) each variable gets.

⚠️ Complexity Alert
Optimal register allocation is NP-complete in the general case. Compilers use heuristics and graph coloring approximations.

Graph Coloring for Register Allocation

This is the most important technique. The idea maps beautifully to a graph-coloring puzzle:

  1. Build a Register Interference Graph (RIG)
    • Each variable = a node in the graph
    • An edge between two variables = they are simultaneously live (both needed at the same time)
  2. Color the graph with k colors, where k = number of physical registers
  3. Two adjacent nodes (variables that interfere) cannot share a color (register)
  4. If k colors suffice → no spilling needed. If not → some variable must be spilled to memory
a and d don't interfere → they can share R1! b→R2, c→R3. Only 3 registers needed.

Register Spilling

When there are more simultaneously live variables than registers, one variable must be spilled — stored in memory. The compiler inserts STORE and LOAD instructions around the spill point. Spilling is expensive, so the compiler tries to spill the variable that's used least frequently.

Global Register Allocation (Loop-Level)

For loops (where programs spend most time), it's worth keeping frequently-used variables in registers for the entire loop duration. The compiler computes a usage count:

Savings(x, loop L) = Σ [use(x, B) + 2 × live(x, B)]  for all blocks B in L

Variables with the highest savings scores get registers first.

5. Evaluation Order & Code Generation

Q: When instructions are independent, is it possible to change evaluation order during code generation? Justify with example. 5 Marks

Yes — when instructions are independent (no data dependency between them), the compiler can freely reorder them to improve efficiency. This is safe because the final result doesn't change.

Why Reordering Helps

Different orderings may require different numbers of temporary registers, reducing register pressure and avoiding memory spills.

Example

Consider: a = b + c; d = e * f; g = a - d

Statements 1 and 2 are independent (neither reads the other's output). The compiler can reorder them.

Original order:           Reordered (may be better):
1. t1 = b + c             1. t1 = e * f   (MUL is slow - start early!)
2. a = t1                 2. t2 = b + c
3. t2 = e * f             3. a = t2
4. d = t2                 4. d = t1
5. g = a - d              5. g = a - d

The reordered version can overlap the multiply (slow) with the add (fast) on a pipelined CPU — effectively hiding the multiply latency.

✅ Key Rule
Reordering is safe only when there is no data dependency between instructions. A dependency exists when one instruction reads the result of another.

6. Runtime Organization

Runtime organization is how the compiler manages memory while the compiled program is executing. Think of it as city planning — different zones (code, stack, heap, static data) are organized into a logical memory map.

Program memory layout during execution
Q: Specify two or more roles of runtime organization in the code generation phase. 5 Marks

Runtime organization plays several critical roles during code generation:

  1. Storage Allocation & Layout: The compiler must decide how to allocate memory for variables. Static allocation (global/static vars at fixed addresses) is decided at compile time. Stack allocation (locals, parameters) uses push/pop at runtime. Heap allocation is for dynamically created objects.
  2. Procedure Call Linkage: Runtime organization defines how function calls work — how parameters are passed, how return values come back, how the return address is stored, and how local variables are accessed. The compiler generates code that sets up and tears down activation records following the calling convention.
  3. Variable Access Mechanism: The compiler generates address-computation code based on where variables live. A local variable x might be accessed as offset_from_frame_pointer (e.g., -8(fp)), while a global is accessed via its absolute address.
  4. Interface to OS & I/O: Runtime organization includes how the program interacts with the operating system for I/O operations, system calls, and exception handling.

7. Activation Records (Stack Frames)

Every time a function is called, the runtime creates a block of memory on the stack called an Activation Record (or stack frame). This stores everything the function needs while it runs.

🧠 Mnemonic — Contents of an Activation Record
RLTAPS

Return value · Link (return address) · Temporaries · Actual parameters · Pointers (control/access links) · Saved registers & local state

The activation record (stack frame) stores everything a function needs to run

Activation Trees

The sequence of function calls forms a tree structure — the Activation Tree. Each node is a call to a function. The path from root to the currently executing node corresponds to the current call stack.

Each function call pushes a frame; return pops it. Activation tree = the structure of calls.

Module 7: Parallelism

Modern computers don't run one instruction at a time — they do many things simultaneously. Module 7 covers how compilers exploit this, from automatic parallelization to vector processing and software pipelining.

1. Types of Parallelism

Flynn's Taxonomy — the standard way to classify parallel architectures

Instruction-Level Parallelism (ILP)

Multiple instructions within the same program are executed in parallel. Examples:

  • Pipelining — fetch, decode, execute stages overlap
  • Out-of-order execution — CPU reorders safe instructions
  • Superscalar — multiple execution units

Data Parallelism

The same operation is applied to many data elements simultaneously. This is what vector processors and GPUs do. Array operations like C[i] = A[i] + B[i] are perfect candidates.

Task-Level Parallelism

Different tasks or threads run concurrently on separate cores. Used in multi-core CPUs. Requires explicit programming with threads (pthreads, OpenMP) or automatic analysis.

2. Automatic Parallelization

Automatic parallelization is when the compiler — not the programmer — converts sequential code into parallel code. This is difficult because the compiler must prove it's safe to run code in parallel.

Loop Parallelization — The Key Case

Most parallelization happens in loops. A loop can be parallelized if its iterations are independent (no iteration depends on the result of a previous one).

// PARALLELIZABLE — each iteration is independent
for (int i = 0; i < N; i++) {
    A[i] = B[i] * C[i];   // A[0] doesn't depend on A[1] etc.
}
// Compiler can emit: #pragma omp parallel for

// NOT PARALLELIZABLE — loop-carried dependency!
for (int i = 1; i < N; i++) {
    A[i] = A[i-1] + 1;    // A[2] needs A[1], A[3] needs A[2]...
                            // Can't run iterations in any order
}
⚠️ The Main Challenge — Data Dependence Analysis
The compiler uses data dependence analysis to find loop-carried dependencies. If the loop body accesses arrays with indices that might alias or overlap, the compiler must be conservative and not parallelize.

Challenges with Irregular Data Structures

Graphs and sparse matrices create irregular memory access patterns. The compiler can't statically determine which elements are accessed, making it nearly impossible to automatically parallelize. This is the topic of a major exam question:

Q: Evaluate the effectiveness of Automatic Parallelization and Cache Locality Optimization in handling complex, real-world applications with irregular data structures (e.g., graphs, sparse matrices). Analyze the challenges compilers face. 10 Marks

Effectiveness on Regular Applications (6 marks)

Automatic parallelization and cache locality optimization are highly effective for regular, structured computations:

  • Dense matrix operations: Matrix multiply has predictable access patterns. Compilers can apply loop tiling, interchange, and vectorization (SIMD) to achieve near-peak performance.
  • Signal processing: FFT, convolution — regular stride access, easily vectorized.
  • Embarrassingly parallel loops: Independent iterations → compiler auto-inserts OpenMP or vectorizes with SSE/AVX instructions.

Challenges with Irregular Data Structures (4 marks)

Graphs and sparse matrices break all the assumptions that make automatic optimization possible:

  1. Unpredictable Memory Access: In a graph traversal (for each neighbor v of u), the memory locations accessed depend on the input data, not a compile-time formula. The compiler cannot compute offsets or predict cache behavior. 
    // Sparse Matrix-Vector: y[i] += val[j] * x[col[j]]
// col[j] is data-dependent → can't know access pattern
  2. Loop-Carried Dependencies: BFS/DFS on a graph has implicit data dependencies between iterations (e.g., the "visited" array). The compiler cannot prove iterations are independent.
  3. Aliasing: Pointer-based graph structures have aliasing — two pointers might reference the same node. The compiler cannot safely parallelize without alias analysis.
  4. Load Imbalance: Even if the compiler parallelizes (e.g., processes different nodes in parallel), nodes with many vs few neighbors create unequal workloads, defeating the speedup.
  5. Cache Locality Failure: Graph traversal follows edges randomly across memory, causing frequent cache misses. Loop tiling and blocking don't apply because there's no spatial locality.
✅ Conclusion
Automatic parallelization is effective for regular computations but largely fails for irregular structures. Real-world high-performance graph processing (like Google's Pregel, GraphX) requires hand-crafted runtime systems, not compiler automation. Compilers can help with inspector-executor techniques or speculation-based parallelism, but these are runtime approaches, not pure compile-time analysis.

3. Cache Locality Optimization & Vectorization

Modern CPUs are fast; memory is slow. The cache bridges this gap — but only if data accesses are nearby in memory (spatial locality) or reused soon (temporal locality).

🧠 Mnemonic — Cache Optimization Techniques
TILUP

Tiling/Blocking · Interchange · Loop Unrolling · Unify (data alignment) · Prefetching

🔲 Loop Tiling (Blocking)

Divide the iteration space into tiles that fit in cache. Instead of sweeping all of a matrix row, process a small block completely before moving on.

// Without tiling (poor cache use for B):
for i: for j: for k:
  C[i][j] += A[i][k] * B[k][j];

// With tiling (B block stays in cache):
for i0 by T: for j0 by T: for k0 by T:
  for i=i0 to i0+T: for j=j0 to j0+T:
    for k=k0 to k0+T:
      C[i][j] += A[i][k] * B[k][j];

🔄 Loop Interchange

Swap the order of nested loops to improve spatial locality. In C/C++, 2D arrays are stored in row-major order — iterating over columns in the inner loop is bad!

// BAD — column-major access (cache misses!)
for (j=0; j<N; j++)
  for (i=0; i<N; i++)
    A[i][j] = ...;   // jumps N elements each step

// GOOD — row-major access (sequential!)
for (i=0; i<N; i++)
  for (j=0; j<N; j++)
    A[i][j] = ...;   // consecutive memory

📦 Loop Unrolling

Replicate the loop body multiple times, reducing loop control overhead and exposing more ILP.

// Original:
for i: A[i] = B[i] + C[i];

// Unrolled by 4:
for i by 4:
  A[i]   = B[i]   + C[i];
  A[i+1] = B[i+1] + C[i+1];
  A[i+2] = B[i+2] + C[i+2];
  A[i+3] = B[i+3] + C[i+3];
// CPU can issue all 4 independently!

🔧 Data Alignment & Prefetching

Alignment: Place array data on cache-line boundaries so one fetch gets all elements of a SIMD register.

Prefetching: Compiler inserts prefetch instructions to load cache lines before they're needed, hiding memory latency.

// Compiler-inserted prefetch hint:
__builtin_prefetch(&A[i+16], 0, 1);
// Load A[i+16] into cache now,
// so it's ready when we get there

4. Scalar vs Vector Processors

Q: Analyse the difference between scalar processor and vector processor with respect to compiler design aspects. Illustrate with relevant examples.
Aspect Scalar Processor Vector Processor
Basic Unit Operates on ONE element at a time Operates on a VECTOR (array) of elements in one instruction
Instruction Style ADD R1, R2, R3 — scalar add VADD V1, V2, V3 — adds 4/8/16 elements at once
Hardware Single ALU Multiple functional units or SIMD lanes
Compiler Strategy Generate one instruction per operation. Focus on register allocation, instruction scheduling. Auto-vectorization: detect loop parallelism, pack data into vector registers, generate SIMD instructions.
Loop Handling Loop runs N iterations sequentially Loop runs N/width iterations (width = vector width, e.g., 8 for 256-bit AVX)
Memory Access One load/store per element Vector load: fetch 8 floats in one instruction
Data Dependency Standard RAW/WAW/WAR checks Must ensure NO aliasing across vector width (harder to verify)
Examples Standard x86 integer ops, early RISC CPUs Intel SSE/AVX, ARM NEON, GPU compute shaders

Concrete Code Example

// C source:
for (int i = 0; i < N; i++)
    C[i] = A[i] + B[i];

// Scalar code (x86):
loop:
  MOV EAX, [A + i*4]
  ADD EAX, [B + i*4]
  MOV [C + i*4], EAX
  INC i; CMP i, N; JLT loop      // N iterations

// Vector code (AVX, 256-bit = 8 floats at once):
loop:
  VMOVAPS YMM0, [A + i*4]   ; load 8 floats from A
  VADDPS  YMM0, YMM0, [B+i*4] ; add 8 floats from B
  VMOVAPS [C + i*4], YMM0   ; store 8 results
  ADD i, 8; CMP i, N; JLT loop  // N/8 iterations!
✅ Compiler Implication
For a vector processor, the compiler's vectorizer must: (1) detect the loop is parallelizable, (2) check alignment, (3) handle the remainder (N mod 8), (4) choose the right SIMD instruction set. This is far more complex than scalar code generation.

5. Instruction Scheduling & Software Pipelining

Even with a single thread, instructions take different numbers of clock cycles. A divide might take 20+ cycles; an add takes 1. If the next instruction needs the divide result, the CPU stalls. Instruction scheduling avoids these stalls.

💡 Intuition — The Traffic Jam Analogy
Think of instructions as cars at a traffic light. If a slow truck (divide) blocks the fast cars behind it, everyone stalls. Instruction scheduling is like building an extra lane — route the fast cars around the truck while it crawls through.

Instruction Scheduling

The compiler reorders instructions (within the constraints of data dependencies) to fill pipeline stalls with useful work.

Moving LOAD R5 between LOAD R1 and ADD fills the stall slot productively

Software Pipelining

Software pipelining is an advanced technique for loops. Instead of finishing one iteration before starting the next, it overlaps iterations — like an assembly line.

Software pipelining overlaps different phases of consecutive iterations, like a factory assembly line
Q: Summarize the impact of instruction scheduling and software pipelining in the design of compilers for parallel processing architecture. 5 Marks

Impact of Instruction Scheduling:

  • Hides memory latency: By moving loads early and placing independent instructions in the stall slots, the compiler hides the memory/functional unit latency from the critical path.
  • Improves pipeline utilization: On an in-order pipelined processor, proper scheduling ensures no functional unit sits idle.
  • Reduces code size impact: Unlike loop unrolling, scheduling doesn't increase code size.
  • Critical for VLIW: Very Long Instruction Word architectures (like DSPs and some compilers for embedded systems) rely entirely on the compiler to schedule instructions correctly, since the hardware doesn't do it.

Impact of Software Pipelining:

  • Maximizes throughput: All functional units (ALU, FPU, load unit) are kept busy in every cycle.
  • Achieves near-peak performance: The initiation interval (II) — minimum cycles between starting consecutive iterations — approaches the theoretical minimum.
  • Essential for SIMD loops: Combined with vectorization, software pipelining provides the maximum possible throughput on superscalar out-of-order processors.
  • Trade-off: Requires a prologue (startup) and epilogue (wind-down) phase, increasing code complexity. The compiler must track "which iteration is each unit processing?"

6. Design of Parallel Machines That Support Parallelizing Compilers

Q: Discuss the design aspects of parallel machines which support parallelizing compilers with respect to instruction streams (5 marks) and architecture (5 marks). Also: Outline the design aspects of parallel machines that support parallelizing compilers with respect to instruction and data streams. 10 Marks

Part A: Design with Respect to Instruction Streams (5 marks)

The way a machine handles instruction streams fundamentally determines how a parallelizing compiler can exploit it:

  1. Multiple Instruction Issue (Superscalar): The processor has multiple execution units and can issue 2–6 instructions per cycle. The compiler's instruction scheduler ensures these units are all fed with independent instructions simultaneously. Example: Intel's Core processors issue up to 6 µops/cycle.
  2. VLIW (Very Long Instruction Word): Each wide instruction word contains multiple "slots" — one per functional unit. The compiler (not hardware) decides how to pack instructions. This shifts scheduling complexity to the compiler but simplifies hardware. Example: Intel Itanium (IA-64).
  3. SIMT (Single Instruction Multiple Threads): As used in GPU architectures — one instruction is broadcast to many "lanes," each handling a different thread. The compiler generates kernel code where each thread has the same instruction sequence but different data.
  4. Instruction-Level Parallelism (ILP) Support: Hardware features like out-of-order execution, branch prediction, and register renaming allow the compiler to generate simpler code while the hardware finds ILP dynamically. The compiler can still help by exposing ILP through scheduling and loop transformations.

Part B: Design with Respect to Architecture (5 marks)

At the architectural level, parallel machine design affects the compiler's ability to parallelize code:

  1. Shared Memory Multiprocessors (SMP/UMA): Multiple cores share a single memory space. The compiler can generate multi-threaded code (OpenMP) where threads communicate via shared variables. Cache coherence protocols (MESI) ensure correctness. The compiler must avoid false sharing (different threads writing to the same cache line).
  2. NUMA (Non-Uniform Memory Access): Memory is physically distributed; local memory is faster. The compiler (or runtime) must consider data placement — putting data close to the thread that uses it. Compilers for NUMA systems generate "first-touch" allocation or affinity hints.
  3. Vector Architecture (Data Parallelism): Register files contain vectors (256–512 bits wide). The compiler's vectorizer transforms scalar loops into vector operations. Architecture provides gather/scatter instructions for non-contiguous memory access, which the compiler exploits for irregular patterns where possible.
  4. Memory Hierarchy Design: Large L1/L2/L3 caches with prefetch support allow the compiler to apply loop tiling aggressively. Wide cache lines (64 bytes typical) reward spatial locality optimizations.
  5. Synchronization Primitives: Hardware compare-and-swap, test-and-set, and memory barriers allow the compiler to implement fine-grained synchronization for parallel sections without OS overhead.

7. Static Single Assignment (SSA) Form

SSA is a property of the IR in which every variable is defined (assigned) exactly once. It's not a machine architecture — it's a form of the IR that makes optimization much easier.

💡 Intuition — Why SSA Helps
Normally, the same variable name can be assigned many times. In SSA, each assignment gets a unique name. This makes it trivial to see which assignment "reaches" any use — because there's only ONE definition per name!

Normal IR (not SSA)

x = 1
x = 2       // redefines x!
y = x + 1   // which x is this?

Hard to track — x has two definitions. Optimization is ambiguous.

SSA Form

x₁ = 1
x₂ = 2
y₁ = x₂ + 1  // clearly uses second def

Each use points to exactly one definition. Dead code (x₁ unused) is obvious.

The Φ (Phi) Function

At join points (where multiple paths merge, e.g., after an if-else), SSA uses a special Φ-function to merge multiple definitions:

// Original code:
if (cond) x = 1; else x = 2;
y = x + 1;

// SSA form:
if (cond) x₁ = 1; else x₂ = 2;
x₃ = Φ(x₁, x₂)   // "whichever path was taken"
y₁ = x₃ + 1;

Benefits of SSA for Optimization

  • Dead code elimination: If x₁ has no uses, remove its definition
  • Constant propagation: If x₁ = 5, replace all uses of x₁ with 5
  • Register allocation: SSA form directly maps to interference graphs
  • Parallelism detection: SSA makes data flow dependencies explicit

8. Domain-Specific Languages (DSLs) and Compilation

A DSL is a language designed for a specific problem domain rather than general-purpose programming. DSLs allow high-level expression of domain problems which a specialized compiler can optimize far better than a general-purpose compiler.

Examples of DSLs

  • SQL — database queries
  • OpenGL/GLSL — graphics shaders
  • Halide — image processing pipelines
  • TensorFlow/XLA — machine learning graphs
  • VHDL/Verilog — hardware design
  • Regex — pattern matching

Why DSL Compilers Excel

  • Domain knowledge reveals semantic constraints a general compiler can't see (e.g., matrix dimensions are always positive)
  • Automatic generation of parallel code (e.g., GPU kernels from numpy-style operations)
  • Whole-program optimization impossible in general compilers
  • Halide separates algorithm from schedule — the same image filter can target CPU cache, GPU, or mobile with just a schedule change

Impact of Language Design on Compiler Optimization

Language design choices profoundly affect what compilers can optimize:

9. Parallelism on Multi-Core: Sorting Example

Q: Consider a program for sorting numbers in C. Discuss and illustrate mechanisms by which we may achieve parallelism on modern micro-processors that support multi-core architectures. [5 marks]. Also, provide an example where parallelism is hard to achieve, with justification. [5 marks]

Part A: Achieving Parallelism — Parallel Merge Sort (5 marks)

Merge sort is naturally parallelizable because the left-half and right-half recursive sorts are completely independent.

#include <omp.h>

void parallel_mergesort(int *arr, int n) {
    if (n <= 1) return;
    int mid = n / 2;

    #pragma omp parallel sections  // compiler directive → two threads
    {
        #pragma omp section
        parallel_mergesort(arr, mid);        // Core 0 sorts left half

        #pragma omp section
        parallel_mergesort(arr + mid, n - mid); // Core 1 sorts right half
    }

    merge(arr, mid, n);  // sequential merge (dependencies here)
}
// Time: O(n log n / p) for p processors — linear speedup!

Other mechanisms:

  • Thread-level parallelism: pthreads or OpenMP divide work across CPU cores
  • Data partitioning: Split array into chunks, sort each chunk on a different core, then merge (like GNU parallel sort)
  • SIMD comparison: Use SSE/AVX to compare 8 integers simultaneously in the merge step
  • Cache-aware: Ensure each thread's chunk fits in its L2 cache to avoid cross-core memory traffic

Part B: Where Parallelism is Hard — Bubble Sort (5 marks)

// Classic bubble sort — hard to parallelize:
for (int i = 0; i < n-1; i++) {
    for (int j = 0; j < n-i-1; j++) {
        if (arr[j] > arr[j+1])
            swap(arr[j], arr[j+1]);
    }
}

Why parallelism is hard here:

  1. Loop-carried dependency: In the inner loop, arr[j] and arr[j+1] overlap — iteration j=2 touches arr[2] and arr[3], while j=3 touches arr[3] and arr[4]. They share arr[3] → a data dependency.
  2. Sequential dependency: Each outer loop iteration depends on the result of the previous — the array state changes each pass.
  3. Race conditions: If multiple threads simultaneously check and swap adjacent elements, they can corrupt data without synchronization.
  4. Overhead exceeds benefit: Even with synchronization, the overhead of locking adjacent elements for O(n²) operations exceeds any parallel speedup.

Note: A variant called Odd-Even Transposition Sort does allow limited parallelism but at the cost of more passes.

  1. Data-dependent access: Graph traversals access memory based on input data (edge lists), making compile-time analysis impossible.
  2. Loop-carried dependencies: BFS/DFS have implicit dependencies (the "visited" set) preventing safe parallel execution.
  3. Pointer aliasing: Graph nodes represented as pointers may alias — the compiler can't prove independence.
  4. Cache unfriendliness: Random memory access patterns (following graph edges) defeat spatial and temporal locality — tiling and blocking don't help.
  5. Load imbalance: High-degree vs low-degree nodes create unequal work per thread, reducing parallel efficiency.

Conclusion: For irregular data, runtime parallelization frameworks (e.g., TBB, Cilk) with inspector-executor models are more effective than static compiler analysis.

PQ1: Explain the concept of register spilling. When does it occur and how does a compiler handle it? 5 Marks

Register spilling occurs when the number of simultaneously live variables at some program point exceeds the number of available physical registers. The compiler must "spill" one variable — temporarily storing it in memory and reloading it when needed.

When it occurs: In loops with many computations, at function call sites (caller-saved registers must be spilled), or in code with many live variables.

How the compiler handles it:

  1. Detect the spill using graph coloring — if the interference graph can't be colored with k colors, spilling is needed.
  2. Select which variable to spill — typically the one with the lowest benefit/cost ratio (infrequently used variables are spilled first).
  3. Insert STORE instructions before the spill point and LOAD instructions at each subsequent use.
  4. Re-run the coloring algorithm — the variable is now split into two: before-spill and after-reload.
; Example spill for variable 'x':
; Before spill point:
  ST  [spill_slot_x], Rx   ; save x to memory

; Before use of x:
  LD  Rx, [spill_slot_x]   ; restore x from memory
  ADD Ry, Rx, Rz           ; use x
PQ2: What is loop tiling? Why is it important for cache performance? Give an example using matrix multiplication. 5 Marks

Loop tiling (also called loop blocking) divides the iteration space of a nested loop into rectangular tiles of size T×T. The key idea is that each tile fits entirely within the cache, so all accesses to the tile hit the cache rather than going to main memory.

Importance: In matrix multiplication, the naive triple loop accesses matrix B in a column-major pattern if the inner loop is over k. This causes a cache miss for every access to B (for large matrices). Tiling ensures a B-tile stays in cache for an entire tile of A.

// Naive matrix multiply — poor cache use for B:
for i: for j: for k:
  C[i][j] += A[i][k] * B[k][j];
// B is accessed column-by-column: B[0][j], B[1][j], ...
// Each B[k][j] is N*sizeof(double) bytes apart — cache miss every time!

// Tiled version (tile size T=64 for 64×64 floats = 16KB in L1):
for i0=0 by T: for j0=0 by T: for k0=0 by T:
  for i=i0 to i0+T:
    for j=j0 to j0+T:
      for k=k0 to k0+T:
        C[i][j] += A[i][k] * B[k][j];
// The T×T tile of B stays in cache for all (i,j) in the tile!
// Cache miss rate drops from O(N³) to O(N³/T)

With T=64 and 8-byte doubles: a 64×64 tile = 32KB — fits in L1 cache. This can provide 5–10× speedup on matrix operations.

PQ3: What is an Activation Record? Describe its structure and explain how stack allocation works for nested function calls. 8 Marks

An Activation Record (stack frame) is a block of memory allocated on the runtime stack each time a function is called. It stores all the information the function needs during execution and is popped when the function returns.

Structure (from higher to lower address):

  1. Return value: Space for the function's return value
  2. Actual parameters: Arguments passed by the caller
  3. Control link: Pointer to the caller's activation record (frame pointer)
  4. Access link: Pointer to the enclosing scope's frame (for nested functions)
  5. Return address: The instruction address to resume after return
  6. Saved registers: Callee-saved registers preserved across the call
  7. Local variables: Space for function's local data
  8. Temporaries: Space for intermediate computations

Stack allocation for nested calls (e.g., main → f → g):

// Call sequence: main() calls f(5), which calls g(3)

Stack state when g() is executing:
High address:
  [main's frame    ] ← oldest frame
  [f(5)'s frame    ] ← f's locals, params, return addr
  [g(3)'s frame    ] ← top of stack (sp points here)
Low address (stack pointer sp)

// When g() returns:
//   1. Return value placed in designated location
//   2. Restore fp to f's frame (from control link)
//   3. Jump to return address
//   4. g's frame is "popped" (sp advances past it)
// Stack is back to just [main, f]

This LIFO (Last In, First Out) discipline matches the natural nesting of function calls perfectly. The frame pointer (fp) provides a stable reference point; local variables are accessed as offset(fp).

PQ4: Explain SSA form and describe how the Φ (phi) function works. Why is SSA useful for optimization? 5 Marks

Static Single Assignment (SSA) form is an IR property where every variable is assigned exactly once. When a variable is reassigned, a new version is created with a subscript (x₁, x₂, etc.).

The Φ (Phi) function is needed at join points — places in the control flow graph where multiple paths merge (after if-else, loop back-edges). The Φ-function takes one argument per incoming path and produces the value that came from whichever path was actually taken at runtime.

// C code:
if (a > b) x = 1; else x = 2;
y = x * 3;

// SSA form:
if (a₁ > b₁) {
  x₁ = 1
} else {
  x₂ = 2
}
x₃ = Φ(x₁, x₂)  // = x₁ if true branch taken, x₂ if false
y₁ = x₃ * 3

Why SSA helps optimization:

  • Dead code elimination: x₁ has no uses → remove it
  • Constant propagation: If x₁ = 5 always, replace all x₁ uses with 5
  • Value numbering: Detect when two expressions always produce the same value
  • Register allocation: Each SSA name = one live range = one node in the interference graph
PQ5: Compare SISD, SIMD, and MIMD architectures. For each, describe how a parallelizing compiler would target it differently. 6 Marks
ArchitectureDescriptionCompiler StrategyExample
SISD Single instruction, single data — purely sequential Standard scalar code gen: instruction scheduling, register allocation, loop unrolling for ILP Classic single-core CPU
SIMD One instruction operates on multiple data elements (vector of width w) Auto-vectorization: detect data-parallel loops, pack elements into vector registers, generate SIMD instructions (SSE/AVX). Handle alignment, remainder loops, and aliasing checks. Intel AVX, ARM NEON, GPU shaders
MIMD Multiple processors execute different instructions on different data independently Parallel code gen: identify parallel loop iterations/sections, generate thread fork/join code (OpenMP), ensure thread-safe memory access, manage synchronization barriers, handle data placement for NUMA Multi-core CPUs, clusters

Key difference for compilers: SIMD vectorization is a within-one-thread optimization; MIMD parallelization is an across-threads optimization. A modern compiler does both: first vectorize inner loops (SIMD), then parallelize outer loops across cores (MIMD). This is called hierarchical parallelism.

PQ6: What is Next-Use Information and how is it computed? Why does the code generator need it? 5 Marks

Next-Use Information tells the code generator whether a variable's current value will be used again in the future. A variable is "live" if it will be used; "dead" if it will not. This is also called liveness analysis.

Why needed: The code generator uses next-use to make register allocation decisions. If a variable is dead (no future use), its register can be immediately freed and reused. If it's live, the value must be preserved.

Algorithm (backwards scan through a basic block):

  1. Initialize all variables as "dead" at block exit (or live if live-out)
  2. Scan statements backwards from last to first
  3. For statement x = y op z:
    • Record next-use info for x, y, z at this point
    • Set x to dead (just defined here — definition kills previous liveness)
    • Set y, z to live (just used)
// Block:         ← backward scan →
(1) a = b + c   x has no next-use after (1)
(2) x = a * d   b,c live after (1); a live (used in 2); d live
(3) e = x + 1   x live (used in 3); e has no later use

Result: at stmt (2), 'a' is live (next use at stmt 2), 'x' is dead at stmt 1
At stmt (3), 'x' is live (used right here), 'e' is dead after
