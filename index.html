1 Introduction to Code Optimization

🧠 Intuition First
Think of code optimization like a GPS that finds not just a route, but the fastest route. The compiler takes your intermediate code and tries to make it run faster and use less memory — without changing what it does.

The code optimization phase improves the intermediate code so that the resulting machine code is faster and smaller. It does not change the meaning of the program — it only changes how it is computed.

Goal: Less time + Less space, same output.
Better intermediate code → Faster machine code

Criteria for Code-Improving Transformations

#CriterionWhat it means
1Meaning-preservingMust not change the output of the program
2Measurably fasterMust provide a demonstrable speed improvement
3Worth the effortCompilation time overhead must be justified

Organization of the Code Optimizer

Types of Optimization

🏠 Local Optimization

Applied within a single basic block. Fastest to compute, done first.

Example: Replacing x=x+0 with nothing.

🌐 Global Optimization

Applied across multiple basic blocks. Requires data flow analysis.

Example: Moving a loop-invariant computation outside a loop.

🔁 1. Common Subexpression Elimination (CSE)
Definition: If an expression was previously computed and the values of its variables have not changed, reuse the result instead of recomputing it.
🧠 Intuition
If you already calculated 4 * i earlier in the same block, why calculate it again? Store it in a temp variable and reuse it!

Two conditions for CSE:

  1. The expression was previously computed.
  2. The values of variables have not changed since that computation.
📝 Example 1 — Simple CSE
a = b + c d = b + c ← same as above! e = a * d
⇒
t = b + c a = t d = t ← reuse t e = a * d

b+c is computed only once. d gets the same value via t.

📝 Example 2 — Classic from Textbook
a = b + c b = a - d c = b + c ← b changed! not CSE d = a - d ← CSE with line 2
⇒
a = b + c b = a - d c = b + c d = b ← reuse b (which = a-d)

⚠️ b+c is NOT a CSE because b changed on line 2. But a-d IS a CSE!

📝 Example 3 — Loop context
t6 := 4*i x := a[t6] t7 := 4*i ← same as t6! t8 := 4*j t9 := a[t8] a[t7] := t9 t10 := 4*j ← same as t8! a[t10] := x
⇒
t6 := 4*i x := a[t6] ← t7 removed t8 := 4*j t9 := a[t8] a[t6] := t9 ← use t6 ← t10 removed a[t8] := x ← use t8
📋 2. Copy Propagation
Definition: After a copy statement f := g, replace subsequent uses of f with g (wherever f is not redefined and g is still available).
🧠 Intuition
If you set f = g, then whenever you see f later, you can just say g instead — one fewer variable to track!
📝 Example
f = g ... y = f * 2 ← f is just g
⇒
f = g ... y = g * 2 ← propagated g
⚠️ Important constraint: Copy propagation is valid only if neither f nor g is redefined between the copy statement and the use point.
💀 3. Dead Code Elimination
Live variable: A variable is live at a point if its value may be used later.
Dead variable: A variable whose value is never used after being assigned.
🧠 Intuition
If you do a calculation but never use the result, that computation is wasted effort — remove it entirely!
📝 Example
x = a + b ← x never used again! y = c * d return y
⇒
← x=a+b deleted! y = c * d return y
🔢 4. Constant Folding
Definition: Evaluate constant expressions at compile time and replace them with their computed values.
🧠 Intuition
Why compute 3.14159 * 2 every time the program runs? The compiler can calculate it once: 6.28318.
📝 Examples
PI = 22/7 area = PI * r * r x = 2 * 3 + 1
⇒
PI = 3.14 ← folded! area = 3.14 * r * r x = 7 ← folded!

3 Basic Blocks & Flow Graphs

Basic Block: A maximal sequence of consecutive statements where control enters at the beginning and exits at the end — no branches in or out in the middle.
🧠 Intuition
A basic block is like a straight road — you enter at one end and drive through without any turns or exits until you reach the end. A flow graph then connects these roads with junctions.

Algorithm to Identify Basic Blocks

Step 1: Find all leaders (first statements of basic blocks):
  ① The first statement of the program is a leader.
  ② Any statement that is the target of a conditional or unconditional goto is a leader.
  ③ Any statement immediately following a goto or conditional goto is a leader.

Step 2: Each leader starts a block that runs until (but not including) the next leader.
📝 Example — Find Basic Blocks
10 i = 0 20 a = n - 3 30 if i < a goto 50 ← goto target: 50 is leader; next stmt (40) is leader 40 goto 170 ← next stmt would be leader 50 b = i - 4 ... 160 if i < a goto 50 ← goto target: 50 already leader 170 ----

Leaders: 10 (first), 40 (after conditional), 50 (target), 170 (target of goto 170)

BlockStatements
B110, 20, 30
B240
B350–160 (loop body)
B4170

Flow Graph

A flow graph connects basic blocks with directed edges — from sequential flow or goto/branch targets.

Algebraic Identities

4 DAG Representation of Basic Blocks

DAG (Directed Acyclic Graph): A graph representation of a basic block where nodes represent values/operations and edges represent dependencies. DAGs detect common subexpressions automatically!
🧠 Intuition
A DAG is like a recipe dependency chart. If two dishes need the same ingredient preparation, you draw one node for that preparation and point both dishes to it — no duplication!

DAG Construction Rules

For each statement x := y op z:
  1. Find or create nodes for y and z
  2. Check if there's already a node for y op z — if yes, that's the CSE node!
  3. If no, create a new internal node labeled op with children y, z
  4. Label this node with x (remove x from any previous node)
📝 Example — Build DAG for basic block

Basic block: a := b+c / b := a-d / c := b+c / d := a-d

StatementActionCSE?
a := b + cCreate node (+) with children b, c. Label it aNo
b := a - dCreate node (-) with children a, d. Label it bNo
c := b + cCreate node (+) with children b(new), c. Label it cNo — b changed!
d := a - dNode (a-d) already exists! Label it d too✅ YES — CSE found!

The node labeled b, d shows that both b and d hold the value of a - d. This is the common subexpression detected by the DAG!

Benefits of DAG

  • Automatically detects common subexpressions
  • Enables dead code elimination (nodes with no outgoing references)
  • Determines which variables are computed in the block
  • Allows reordering of independent computations

5 Peephole Optimization

Peephole Optimization: A local optimization technique that examines a small "window" (peephole) of target code instructions and replaces inefficient instruction sequences with better ones.
🧠 Intuition
Imagine proofreading a document through a magnifying glass — you see only a few lines at a time, but you spot and fix local redundancies. Peephole optimization does the same for machine code!
🔑 Mnemonic: "R-FAS-UR"
Redundant Instruction Elimination · Flow-of-control Optimization · Algebraic Simplification · Strength Reduction · Use of Machine Idioms · Remove Unreachable Code
1️⃣ Redundant Instruction Elimination
MOV R0, a ← store a into R0 MOV a, R0 ← store R0 back (REDUNDANT!)
⇒
MOV R0, a ← keep only this second instruction deleted!

The second instruction reloads what was just stored. Since R0 already has the value of a, the second MOV is useless.

2️⃣ Flow-of-Control Optimization
goto L1 ... L1: goto L2 ← chain of jumps
⇒
goto L2 ← skip to final target ... L1: goto L2

Eliminate jump chains — jump directly to the final destination.

3️⃣ Algebraic Simplification
x = x + 0 ← useless y = y * 1 ← useless z = z ** 1 ← useless
⇒
← all three deleted!
4️⃣ Use of Machine Idioms
a = a + 1 ← generic add
⇒
INC a ← hardware increment!

Modern CPUs have special instructions for common patterns like increment/decrement.

5️⃣ Unreachable Code Elimination
#define DEBUG 0 ... if(DEBUG) print("info") ← never runs!
⇒
← entire debug print removed!

After constant propagation substitutes DEBUG=0, the condition becomes always false — the block is unreachable and deleted.

6️⃣ Reduction in Strength (Peephole)
x = y ** 2 ← function call!
⇒
x = y * y ← single multiply!

6 Loops in Flow Graphs

🧠 Why loops matter most?
A loop body that runs 10,000 times — even a tiny saving per iteration saves 10,000 operations! This is why loop optimization gives the biggest performance gains.

Dominators

Dominator: Node d dominates node n (written d dom n) if every path from the entry node to n passes through d.
Back Edge: An edge a → b where b dominates a. Back edges identify loops!

Two essential properties of a loop:
1. Single entry point called the header (dominates all nodes in the loop)
2. At least one way to iterate (path back to header)
🚀 Loop Optimization 1: Code Motion
Code Motion: Move a loop-invariant computation (one that gives the same result regardless of loop iterations) to the preheader — before the loop.
🧠 Intuition
If you're computing the same thing every iteration (like limit - 2), compute it once before the loop and store it.
📝 Example
while (i <= limit - 2) { // loop body i++; }
⇒
t = limit - 2; ← moved out! while (i <= t) { // loop body i++; }

limit - 2 is computed once instead of every iteration. If the loop runs 1000 times → we save 999 subtractions!

Condition: The expression must be loop-invariant — none of its variables change value inside the loop.
📉 Loop Optimization 2: Induction Variables & Reduction in Strength
Induction Variable: A variable that changes by a constant amount each loop iteration (e.g., i = i + 1). Any variable linearly related to an induction variable is also an induction variable.
Reduction in Strength: Replace an expensive operation (like multiplication) with a cheaper one (like addition) by exploiting the induction variable relationship.
🧠 Intuition
If i increases by 1 each time, then 4*i increases by 4 each time. Instead of doing t = 4*i (a multiply) every iteration, maintain t directly by adding 4: t = t + 4.
📝 Example — Strength Reduction on 4*j
j := j - 1 t4 := 4 * j ← multiply every iter! t5 := a[t4] if t5 > v goto B3
⇒
j := j - 1 t4 := t4 - 4 ← just subtract! t5 := a[t4] if t5 > v goto B3
TechniqueWhat changesSaving
Code MotionLoop-invariant expr moved outN × (cost of expr)
Strength ReductionMultiply → Add per iterationN × (multiply - add cost)
Induction Var Elim.Remove redundant variablesN × (var update cost)

7 Introduction to Data Flow Analysis

Data Flow Analysis: A framework for collecting information about the flow of data values through a program. Used to detect reaching definitions, live variables, available expressions, etc.
🧠 Intuition
Think of data flow analysis like tracking water through pipes. At any point in the program, which "definitions" (assignments) of a variable can "flow" to reach that point without being "blocked" (overwritten)?

Reaching Definitions

A definition d of variable x reaches a point p if there exists a path from d to p along which x is not redefined.

Data Flow Equation

The fundamental data flow equation:

out[S] = gen[S] ∪ (in[S] − kill[S])
TermMeaning
out[S]Definitions leaving statement S
gen[S]Definitions generated by S itself
in[S]Definitions arriving at S from predecessors
kill[S]Definitions of the same variable killed by S

Bit Vector Representation

🧠 Intuition
Number all definitions in the program (d1, d2, d3 ...). Use a bit vector where bit i = 1 means definition di is in the set. Set union = bitwise OR. Set intersection = bitwise AND. Very efficient!
📝 Example
4 definitions: d1, d2, d3, d4 gen[B1] = {d1, d3} → bit vector: 1 0 1 0 kill[B1]= {d2, d4} → bit vector: 0 1 0 1 Union (OR): 1 0 1 0 | 0 1 0 1 = 1 1 1 1 Intersect(AND): 1 0 1 0 & 0 1 0 1 = 0 0 0 0

📋 Q1 — 2 Marks · Definition

What is code optimization? What are the criteria for code-improving transformations?

A
Code Optimization is the process of improving intermediate code so that the resulting machine code runs faster and/or uses less memory, without changing the program's output.

Criteria: (1) Meaning-preserving — must not change what the program computes. (2) Measurably faster — must demonstrably speed up execution. (3) Worth the effort — compilation overhead must be justified.
📋 Q2 — 2 Marks · Definition

Define: (a) Basic Block (b) Dead Code (c) Induction Variable (d) Loop Invariant Computation

A

(a) Basic Block: A maximal sequence of consecutive three-address instructions where control enters at the beginning and leaves at the end without halt or possibility of branching.

(b) Dead Code: A statement that computes a value that is never subsequently used. The variable assigned is "dead" — its value will never be read.

(c) Induction Variable: A variable that changes by a constant amount every loop iteration. Example: i = i + 1.

(d) Loop Invariant Computation: An expression inside a loop whose value does not change across iterations. Example: limit - 2 where limit is not modified in the loop.

Explain the data flow equation out[S] = gen[S] ∪ (in[S] − kill[S]) with an example.

📋 Q12 — 2 Marks · Concept

What are dominators in a flow graph? Why are they important?

A

Node d dominates node n if every possible path from the entry node to n must pass through d. Written as d dom n.

Importance:

  • Used to identify natural loops (back edges: n→d where d dominates n)
  • The loop header dominates all nodes in the loop
  • Used in code motion to verify a computation dominates all loop exits
  • Foundation for reducible flow graph analysis

Key fact: Every node dominates itself.

6 Module 6: Code Generation

The final phase of compilation — translating optimized IR into target machine code

Code generation is the final phase of compilation. The compiler takes the optimized intermediate representation (IR) and translates it into target machine code.

Compiler pipeline — Code Generator is the final translation step

1. Issues in the Design of a Code Generator

🧠 Mnemonic — Remember the 5 Issues
I T R E O

Input (IR format) · Target program (architecture) · Register allocation · Evaluation order · Optimal code generation

1. Input to Code Generator

Receives the IR + symbol table. Common IRs: three-address code, DAG, bytecode.

2. Target Program

Output can be: absolute machine code, relocatable object code, or assembly code.

3. Register Allocation

Deciding which values stay in registers vs memory. NP-complete in the general case.

4. Evaluation Order

The order of computing operations affects how many temporary registers are needed.

2. Target Machine Architectures

RISC — Reduced Instruction Set Computer

  • Many registers (typically 32+)
  • All instructions are 3-address: ADD R1, R2, R3
  • Fixed-length instructions (e.g., 32 bits)
  • Examples: ARM, MIPS, RISC-V
# RISC: a = b + c
LD  R1, b       ; Load b into R1
LD  R2, c       ; Load c into R2
ADD R3, R1, R2  ; R3 = R1 + R2
ST  a, R3       ; Store result to a

CISC — Complex Instruction Set Computer

  • Few registers, but instructions can access memory directly
  • 2-address instructions: destination is also a source
  • Examples: x86, x86-64
; CISC (x86): a = b + c
MOV EAX, b    ; EAX = b
ADD EAX, c    ; EAX = EAX + c
MOV a, EAX   ; a = EAX

Stack-Based Machine

  • Operations work on a stack — push operands, operation pops and pushes result
  • Java Virtual Machine (JVM) is a famous example
// Stack: a = b + c
PUSH b   ; stack: [b]
PUSH c   ; stack: [b, c]
IADD     ; pops b,c, pushes (b+c)
POP a    ; a = b+c

3. Next-Use Information

💡 Intuition — Why Next-Use Matters
If you won't need what's in a register again, you can free it. Next-use info tells the code generator when it's safe to free a register.

Algorithm: Computing Next-Use (backwards scan)

  1. Initially, all variables are "dead" (no next-use).
  2. When you see an assignment x = y op z: record current next-use info, mark x as dead, mark y, z as live.

Example

Statement   | Variable a | Variable b | Variable c
-----------+------------+------------+------------
(1) a = b+c | dead (def) | use at 1   | use at 1
(2) d = a-b | live@2     | dead (def) | live@1
(3) e = b+c | dead       | use@2,3    | use@1,3

4. Register Allocation and Assignment

Two Sub-Problems

Register Allocation — Decide which variables get to live in registers at each program point.

Register Assignment — Decide which specific register (R1, R2, R3...) each variable gets.

⚠️ Complexity Alert
Optimal register allocation is NP-complete in the general case. Compilers use heuristics and graph coloring approximations.

Graph Coloring for Register Allocation

  1. Build a Register Interference Graph (RIG) — each variable = a node; edge = two variables live at same time
  2. Color the graph with k colors (k = number of physical registers)
  3. Adjacent nodes (interfering variables) cannot share a color (register)
  4. If k colors suffice → no spilling. If not → some variable must be spilled to memory
a and d don't interfere → they can share R1! b→R2, c→R3. Only 3 registers needed.

6. Runtime Organization

Runtime organization is how the compiler manages memory while the compiled program is executing — different zones (code, stack, heap, static data) organized into a logical memory map.

Program memory layout during execution
Q: Specify two or more roles of runtime organization in the code generation phase. 5 Marks
  1. Storage Allocation & Layout: Compiler decides how to allocate memory — static allocation (globals at fixed addresses), stack allocation (locals via push/pop), heap allocation (dynamic objects).
  2. Procedure Call Linkage: Defines how function calls work — how parameters are passed, how return values come back, how the return address is stored, and how local variables are accessed.
  3. Variable Access Mechanism: Generates address-computation code based on where variables live. A local variable x might be accessed as -8(fp), while a global is accessed via its absolute address.
  4. Interface to OS & I/O: Runtime organization includes how the program interacts with the OS for I/O operations, system calls, and exception handling.

Every time a function is called, the runtime creates a block of memory on the stack called an Activation Record (or stack frame).

7 Module 7: Parallelism

How compilers exploit modern hardware — automatic parallelization, vectorization, software pipelining, and SSA

1
Types of Parallelism

Modern computers don't run one instruction at a time — they do many things simultaneously. Module 7 covers how compilers exploit this.

Flynn's Taxonomy — the standard way to classify parallel architectures

Instruction-Level Parallelism (ILP)

Multiple instructions within the same program executed in parallel. Examples: pipelining, out-of-order execution, superscalar.

Data Parallelism

The same operation applied to many data elements simultaneously. Used by vector processors and GPUs. Perfect for array operations like C[i] = A[i] + B[i].

Task-Level Parallelism

Different tasks or threads run concurrently on separate cores. Requires explicit programming with threads (pthreads, OpenMP) or automatic analysis.

2. Automatic Parallelization

Automatic parallelization is when the compiler — not the programmer — converts sequential code into parallel code.

Parallel Code
// PARALLELIZABLE — each iteration is independent
for (int i = 0; i < N; i++) {
    A[i] = B[i] * C[i];   // A[0] doesn't depend on A[1] etc.

// NOT PARALLELIZABLE — loop-carried dependency!

⚠️ The Main Challenge — Data Dependence Analysis
The compiler uses data dependence analysis to find loop-carried dependencies. If the loop body accesses arrays with indices that might alias or overlap, the compiler must be conservative.
Q: Evaluate the effectiveness of Automatic Parallelization and Cache Locality Optimization in handling complex, real-world applications with irregular data structures (e.g., graphs, sparse matrices). 10 Marks

Effectiveness on Regular Applications (6 marks)

Automatic parallelization is highly effective for regular, structured computations:

  • Dense matrix operations: Matrix multiply has predictable access patterns. Compilers apply loop tiling, interchange, and vectorization to achieve near-peak performance.
  • Signal processing: FFT, convolution — regular stride access, easily vectorized.
  • Embarrassingly parallel loops: Independent iterations → compiler auto-inserts OpenMP or vectorizes with SSE/AVX.

Challenges with Irregular Data Structures (4 marks)

  1. Unpredictable Memory Access: In graph traversal, memory locations accessed depend on input data, not a compile-time formula.
  2. Loop-Carried Dependencies: BFS/DFS has implicit data dependencies between iterations (e.g., the "visited" array).
  3. Aliasing: Pointer-based graph structures may alias — compiler cannot safely parallelize.
  4. Load Imbalance: High-degree vs low-degree nodes create unequal workloads.
  5. Cache Locality Failure: Graph traversal follows edges randomly across memory — tiling and blocking don't apply.
✅ Conclusion
Automatic parallelization works well for regular computations but largely fails for irregular structures. Real-world graph processing requires hand-crafted runtime systems.

3. Cache Locality Optimization & Vectorization

🧠 Mnemonic — Cache Optimization Techniques
TILUP

Tiling/Blocking · Interchange · Loop Unrolling · Unify (data alignment) · Prefetching

🔲 Loop Tiling (Blocking)

Divide the iteration space into tiles that fit in cache.

// With tiling (B block stays in cache):
for i0 by T: for j0 by T: for k0 by T:
  for i=i0 to i0+T:
    for k=k0 to k0+T:
      C[i][j] += A[i][k] * B[k][j];

🔄 Loop Interchange

Swap the order of nested loops to improve spatial locality. In C/C++, 2D arrays are stored in row-major order.

// GOOD — row-major access (sequential!)
for (i=0; i<N; i++)
  for (j=0; j<N; j++)
    A[i][j] = ...;

📦 Loop Unrolling

Replicate the loop body multiple times, reducing loop control overhead and exposing more ILP.

🔧 Prefetching

Compiler inserts prefetch instructions to load cache lines before they're needed, hiding memory latency.

Concrete Code Example

// C source:
for (int i = 0; i < N; i++)
    C[i] = A[i] + B[i];

// Scalar code (x86):
loop:
  MOV EAX, [A + i*4]
  ADD EAX, [B + i*4]
  MOV [C + i*4], EAX
  INC i; CMP i, N; JLT loop      // N iterations

// Vector code (AVX, 256-bit = 8 floats at once):
loop:
  VMOVAPS YMM0, [A + i*4]   ; load 8 floats from A
  VADDPS  YMM0, YMM0, [B+i*4] ; add 8 floats from B
  VMOVAPS [C + i*4], YMM0   ; store 8 results
  ADD i, 8; CMP i, N; JLT loop  // N/8 iterations!

5. Instruction Scheduling & Software Pipelining

💡 Intuition — The Traffic Jam Analogy
Think of instructions as cars at a traffic light. If a slow truck (divide) blocks the fast cars behind it, everyone stalls. Instruction scheduling is like building an extra lane — route the fast cars around the truck while it crawls through.

Instruction Scheduling

The compiler reorders instructions (within the constraints of data dependencies) to fill pipeline stalls with useful work.

Moving LOAD R5 between LOAD R1 and ADD fills the stall slot productively
Q: Summarize the impact of instruction scheduling and software pipelining in the design of compilers for parallel processing architecture. 5 Marks

Impact of Instruction Scheduling:

  • Hides memory latency: Moving loads early and placing independent instructions in the stall slots hides the memory/functional unit latency.
  • Improves pipeline utilization: On an in-order pipelined processor, proper scheduling ensures no functional unit sits idle.
  • Critical for VLIW: Very Long Instruction Word architectures rely entirely on the compiler to schedule instructions correctly.

Impact of Software Pipelining:

  • Maximizes throughput: All functional units (ALU, FPU, load unit) are kept busy in every cycle.
  • Achieves near-peak performance: The initiation interval (II) approaches the theoretical minimum.
  • Trade-off: Requires a prologue (startup) and epilogue (wind-down) phase, increasing code complexity.

7. Static Single Assignment (SSA) Form

SSA is a property of the IR in which every variable is defined (assigned) exactly once. It makes optimization much easier.

💡 Intuition — Why SSA Helps
Normally, the same variable name can be assigned many times. In SSA, each assignment gets a unique name. This makes it trivial to see which assignment "reaches" any use — because there's only ONE definition per name!

Normal IR (not SSA)

x = 1
x = 2       // redefines x!
y = x + 1   // which x is this?

Hard to track — x has two definitions. Optimization is ambiguous.

SSA Form

x₁ = 1
x₂ = 2
y₁ = x₂ + 1  // clearly uses second def

Each use points to exactly one definition. Dead code (x₁ unused) is obvious.

The Φ (Phi) Function

// Original code:
if (cond) x = 1; else x = 2;
y = x + 1;

// SSA form:
if (cond) x₁ = 1; else x₂ = 2;
x₃ = Φ(x₁, x₂)   // "whichever path was taken"
y₁ = x₃ + 1;

Benefits of SSA for Optimization

  • Dead code elimination: If x₁ has no uses, remove its definition
  • Constant propagation: If x₁ = 5, replace all uses of x₁ with 5
  • Register allocation: SSA form directly maps to interference graphs
  • Parallelism detection: SSA makes data flow dependencies explicit

Sample Practice Questions & Answers
PQ1: Explain the concept of register spilling. When does it occur and how does a compiler handle it? 5 Marks

Register spilling occurs when the number of simultaneously live variables exceeds the number of available physical registers. The compiler must "spill" one variable — temporarily storing it in memory and reloading it when needed.

When it occurs: In loops with many computations, at function call sites, or in code with many live variables.

How the compiler handles it:

  1. Detect the spill using graph coloring — if the interference graph can't be colored with k colors, spilling is needed.
  2. Select which variable to spill — typically the one with the lowest benefit/cost ratio.
  3. Insert STORE instructions before the spill point and LOAD instructions at each subsequent use.
  4. Re-run the coloring algorithm — the variable is now split into two.
; Example spill for variable 'x':
; Before spill point:
  ST  [spill_slot_x], Rx   ; save x to memory
; Before use of x:
  LD  Rx, [spill_slot_x]   ; restore x from memory
  ADD Ry, Rx, Rz           ; use x
PQ2: What is loop tiling? Why is it important for cache performance? Give an example using matrix multiplication. 5 Marks

Loop tiling (loop blocking) divides the iteration space of a nested loop into rectangular tiles of size T×T. Each tile fits entirely within the cache, so all accesses to the tile hit the cache rather than going to main memory.

// Naive matrix multiply — poor cache use for B:
for i: for j: for k:
  C[i][j] += A[i][k] * B[k][j];

// Tiled version (tile size T=64):
for i0=0 by T: for j0=0 by T: for k0=0 by T:
  for i=i0 to i0+T:
    for j=j0 to j0+T:
      for k=k0 to k0+T:
        C[i][j] += A[i][k] * B[k][j];
// The T×T tile of B stays in cache for all (i,j)!

With T=64 and 8-byte doubles: a 64×64 tile = 32KB — fits in L1 cache. This can provide 5–10× speedup.

PQ3: What is an Activation Record? Describe its structure and explain how stack allocation works for nested function calls. 8 Marks

An Activation Record (stack frame) is a block of memory allocated on the runtime stack each time a function is called. It stores all the information the function needs during execution.

Structure (from higher to lower address):

  1. Return value: Space for the function's return value
  2. Actual parameters: Arguments passed by the caller
  3. Control link: Pointer to the caller's activation record (frame pointer)
  4. Access link: Pointer to the enclosing scope's frame (for nested functions)
  5. Return address: The instruction address to resume after return
  6. Saved registers: Callee-saved registers preserved across the call
  7. Local variables: Space for function's local data
  8. Temporaries: Space for intermediate computations
// Call sequence: main() calls f(5), which calls g(3)
Stack state when g() is executing:
High address:
  [main's frame    ] ← oldest frame
  [f(5)'s frame    ] ← f's locals, params, return addr
  [g(3)'s frame    ] ← top of stack (sp points here)
Low address (stack pointer sp)

This LIFO discipline matches the natural nesting of function calls. The frame pointer (fp) provides a stable reference point; local variables are accessed as offset(fp).

PQ4: Explain SSA form and describe how the Φ (phi) function works. Why is SSA useful for optimization? 5 Marks

Static Single Assignment (SSA) form is an IR property where every variable is assigned exactly once. When a variable is reassigned, a new version is created with a subscript (x₁, x₂, etc.).

The Φ (Phi) function is needed at join points — places where multiple paths merge (after if-else, loop back-edges). It takes one argument per incoming path and produces the value from whichever path was actually taken at runtime.

// C code:
if (a > b) x = 1; else x = 2;
y = x * 3;

// SSA form:
if (a₁ > b₁) {
  x₁ = 1
} else {
  x₂ = 2
}
x₃ = Φ(x₁, x₂)  // = x₁ if true, x₂ if false
y₁ = x₃ * 3

Why SSA helps: Dead code elimination (x₁ with no uses → remove it), constant propagation (x₁ = 5 → replace all x₁ uses with 5), value numbering, and register allocation.

PQ5: Compare SISD, SIMD, and MIMD architectures. For each, describe how a parallelizing compiler would target it differently. 6 Marks
ArchitectureDescriptionCompiler StrategyExample
SISD Single instruction, single data — purely sequential Standard scalar code gen: instruction scheduling, register allocation, loop unrolling for ILP Classic single-core CPU
SIMD One instruction operates on multiple data elements (vector of width w) Auto-vectorization: detect data-parallel loops, pack elements into vector registers, generate SIMD instructions (SSE/AVX). Handle alignment, remainder loops, and aliasing checks. Intel AVX, ARM NEON, GPU shaders
MIMD Multiple processors execute different instructions on different data independently Parallel code gen: identify parallel loop iterations/sections, generate thread fork/join code (OpenMP), ensure thread-safe memory access, manage synchronization barriers Multi-core CPUs, clusters

Key difference: SIMD vectorization is a within-one-thread optimization; MIMD parallelization is an across-threads optimization. A modern compiler does both — this is called hierarchical parallelism.

PQ6: What is Next-Use Information and how is it computed? Why does the code generator need it? 5 Marks

Next-Use Information tells the code generator whether a variable's current value will be used again in the future. A variable is "live" if it will be used; "dead" if it will not.

Why needed: The code generator uses next-use to make register allocation decisions. If a variable is dead (no future use), its register can be immediately freed. If it's live, the value must be preserved.

Algorithm (backwards scan through a basic block):

  1. Initialize all variables as "dead" at block exit
  2. Scan statements backwards from last to first
  3. For statement x = y op z: record next-use info; set x to dead; set y, z to live
// Block:         ← backward scan →
(1) a = b + c   x has no next-use after (1)
(2) x = a * d   b,c live after (1); a live (used in 2); d live
(3) e = x + 1   x live (used in 3); e has no later use

Q2: Discuss the design aspects of parallel machines which support parallelizing compilers with respect to instruction streams (5 marks) and architecture (5 marks). 10 Marks

Part A: Design with Respect to Instruction Streams (5 marks)

  1. Multiple Instruction Issue (Superscalar): The processor has multiple execution units and can issue 2–6 instructions per cycle. The compiler's instruction scheduler ensures all units are fed with independent instructions simultaneously.
  2. VLIW (Very Long Instruction Word): Each wide instruction word contains multiple "slots" — one per functional unit. The compiler (not hardware) decides how to pack instructions. Example: Intel Itanium (IA-64).
  3. SIMT (Single Instruction Multiple Threads): As used in GPU architectures — one instruction is broadcast to many "lanes," each handling a different thread.
  4. ILP Support: Hardware features like out-of-order execution, branch prediction, and register renaming allow the compiler to generate simpler code while the hardware finds ILP dynamically.

Part B: Design with Respect to Architecture (5 marks)

  1. Shared Memory Multiprocessors (SMP/UMA): Multiple cores share a single memory space. The compiler generates multi-threaded code (OpenMP) where threads communicate via shared variables.
  2. NUMA (Non-Uniform Memory Access): Memory is physically distributed; local memory is faster. Compilers for NUMA generate "first-touch" allocation or affinity hints.
  3. Vector Architecture (Data Parallelism): Register files contain vectors (256–512 bits wide). The compiler's vectorizer transforms scalar loops into vector operations.
  4. Memory Hierarchy Design: Large L1/L2/L3 caches with prefetch support allow the compiler to apply loop tiling aggressively.
  5. Synchronization Primitives: Hardware compare-and-swap and memory barriers allow fine-grained synchronization for parallel sections without OS overhead.
Q3(i): When instructions are independent, is it possible to change the evaluation order during code generation? Justify with example. 5 Marks

Yes — when instructions are independent (no data dependency between them), the compiler can freely reorder them. This is safe because the final result doesn't change.

Original order:           Reordered (may be better):
1. t1 = b + c             1. t1 = e * f   (MUL is slow - start early!)
2. a = t1                 2. t2 = b + c
3. t2 = e * f             3. a = t2
4. d = t2                 4. d = t1
5. g = a - d              5. g = a - d

The reordered version can overlap the multiply (slow) with the add (fast) on a pipelined CPU — effectively hiding the multiply latency.

✅ Key Rule
Reordering is safe only when there is no data dependency between instructions.
Q5: Consider a sorting program in C. Discuss mechanisms for parallelism on modern multi-core processors. [5 marks]. Also provide an example where parallelism is hard to achieve. [5 marks]

Part A: Achieving Parallelism — Parallel Merge Sort (5 marks)

Merge sort is naturally parallelizable because the left-half and right-half recursive sorts are completely independent.

#include <omp.h>
void parallel_mergesort(int *arr, int n) {
    if (n <= 1) return;
    int mid = n / 2;
    #pragma omp parallel sections  // two threads
    {
        #pragma omp section
        parallel_mergesort(arr, mid);        // Core 0 sorts left half
        #pragma omp section
        parallel_mergesort(arr + mid, n - mid); // Core 1 sorts right half
    }
    merge(arr, mid, n);  // sequential merge
}
// Time: O(n log n / p) for p processors!

Part B: Where Parallelism is Hard — Bubble Sort (5 marks)

for (int i = 0; i < n-1; i++) {
    for (int j = 0; j < n-i-1; j++) {
        if (arr[j] > arr[j+1])
            swap(arr[j], arr[j+1]);
    }
}

Why parallelism is hard:

  1. Loop-carried dependency: Iteration j=2 touches arr[2] and arr[3], while j=3 touches arr[3] and arr[4]. They share arr[3] — a data dependency.
  2. Sequential dependency: Each outer loop iteration depends on the result of the previous one.
  3. Race conditions: Multiple threads simultaneously checking and swapping adjacent elements can corrupt data.
  4. Overhead exceeds benefit: The overhead of locking adjacent elements for O(n²) operations exceeds any parallel speedup.
