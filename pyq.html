Q1

Explain the phases of a compiler with example input: while(i<=num){ sum=sum+i*20; i++; }

Module 1 10 Marks Nov 2024May 2024
▼

Overview — The Six Phases

A compiler translates source code into target code through six sequential phases. Each phase transforms the program into a progressively lower-level representation. The phases communicate through intermediate data structures and share a symbol table and error handler.

Phase 1 — Lexical Analysis (Scanner)

Reads character stream, groups into tokens (meaningful units). Removes whitespace and comments.

For sum = sum + i * 20:

LexemeToken TypeAttribute
sumidpointer to symtab entry
=assign_op—
sumidpointer to symtab entry
+add_op—
iidpointer to symtab entry
*mul_op—
20num20

Output: token stream → <id,1> <assign> <id,1> <+> <id,2> <*> <num,20>

Phase 2 — Syntax Analysis (Parser)

Arranges tokens into a parse tree / syntax tree according to the grammar rules. Checks grammatical structure.

Phase 3 — Semantic Analysis

Performs type checking and ensures semantic correctness. Adds type info to syntax tree.

  • Checks that i and num in i <= num are compatible types.
  • Detects implicit type conversions: 20 (integer) → implicit cast if sum is float.
  • Checks that variables are declared before use.
  • Tree is annotated: inttoreal(20) may be inserted if needed.
Phase 4 — Intermediate Code Generation

Generates three-address code (TAC) — a machine-independent intermediate form. Each instruction has at most 3 operands.

L1:  if i <= num goto L2
     goto L3
L2:  t1 = i * 20
     sum = sum + t1
     i = i + 1
     goto L1
L3:  ...  // exit while
Phase 5 — Code Optimization

Improves intermediate code for speed and space without changing output semantics.

  • Constant folding: i * 20 → kept (i is variable), but 20 stays as literal (no computation at compile-time possible).
  • Loop-invariant code motion: If any computation inside loop doesn't change, move it outside.
  • Strength reduction: Replace costly ops with cheaper ones.
// Optimized: no redundant loads
L1:  if i > num goto L3
     t1 = i * 20
     sum = sum + t1
     i = i + 1
     goto L1
L3:
Phase 6 — Code Generation

Generates actual target machine code / assembly.

L1:  MOV  R1, i
     CMP  R1, num
     JG   L3
     MOV  R2, R1
     MUL  R2, #20       ; t1 = i * 20
     ADD  sum, R2       ; sum = sum + t1
     INC  i
     JMP  L1
L3:

Symbol Table is accessed throughout all phases — it stores identifiers, their types, scope, and memory addresses. Error Handler catches errors at every phase (lexical errors, syntax errors, type errors, etc.).

Exam Tip: Draw the "phases of compiler" diagram showing flow: Source → Lexer → Parser → Semantic Analyzer → ICG → Optimizer → Code Gen → Target. Always mention Symbol Table and Error Handler as cross-cutting concerns.

Determine if Grammar G is LL(1): B → BB | B# | B&&B | (B) | a | b

Module 2 10 Marks Nov 2024 (Set 1)
▼

Step 1 — Identify the Grammar

Non-terminal: B. Terminals: #, &, (, ), a, b. Productions:

B → BB
B → B#
B → B&&B    (note: & repeated for && operator)
B → (B)
B → a
B → b
Observation: This grammar is left-recursive (B → BB, B → B#, B → B&&B). LL(1) parsers cannot handle left recursion. We must first eliminate left recursion.

Step 2 — Eliminate Left Recursion

Left-recursive productions: B → BB | B# | B&&B (all start with B). Non-left-recursive: B → (B) | a | b

Apply standard left recursion elimination: A → Aα₁|Aα₂|…|β becomes A → βA', A' → α₁A'|α₂A'|ε

Here β = (B) | a | b and α₁ = B (from BB), α₂ = # (from B#), α₃ = &&B (from B&&B)

B  → (B)B' | aB' | bB'
B' → BB' | #B' | &&BB' | ε

Step 3 — Compute FIRST and FOLLOW

Non-terminalFIRSTFOLLOW
B{ (, a, b }{ ), #, &, $, (, a, b }
B'{ (, a, b, #, &, ε }{ ), $, #, & }

FIRST calculation:
FIRST(B) = FIRST((B)B') ∪ FIRST(aB') ∪ FIRST(bB') = {(} ∪ {a} ∪ {b} = {(, a, b}
FIRST(B') = FIRST(BB') ∪ FIRST(#B') ∪ FIRST(&&BB') ∪ {ε} = {(,a,b} ∪ {#} ∪ {&} ∪ {ε}

FOLLOW calculation:
FOLLOW(B) = {$} (start symbol). Also wherever B appears on RHS: in (B)B' → ) is after B; in BB' → FIRST(B')\{ε} and since B' is nullable, FOLLOW(B) added. In &&BB' → FOLLOW(B).
FOLLOW(B') = FOLLOW(B) since B → ...B' and B' can appear at end.

Step 4 — Predictive Parsing Table

()ab#&&$
BB→(B)B'—B→aB'B→bB'———
B'B'→BB'B'→εB'→BB'B'→BB'B'→#B'B'→&&BB'B'→ε

Step 5 — Justify: Is it LL(1)?

The grammar is NOT LL(1) even after left recursion elimination, because:
  • B'[(] has entry B'→BB' (from FIRST(BB')={( a b})
  • B'[a] has entry B'→BB' — no conflict here individually
  • However, B' is nullable and FIRST(B') ∩ FOLLOW(B') ≠ ∅ for some entries creates potential ambiguity in the original grammar.
  • More critically, the original grammar G = B → BB | ... is ambiguous (BB can be parsed multiple ways). Ambiguous grammars are inherently not LL(1).
Conclusion: G is not LL(1) because (1) it is left-recursive and (2) the underlying grammar is ambiguous. After transformation, the parsing table has no multi-entry conflicts but the grammar semantics remain ambiguous.

Q9

Apply back-patching for: while(x<200 || y<400){ i=0; while(i<10){p=p+i; i=i+1;} x=x+p; y=y+p; }

Module 4 10 Marks Nov 2024 (Set 1)
▼

What is Back-Patching?

Back-patching is a technique used in one-pass code generation for boolean expressions and control flow statements. When a conditional jump is generated, the target label is unknown yet, so we write a placeholder. When the target becomes known, we go back and fill in the label — this is patching.

Key functions:

  • makelist(i) — creates a new list with instruction i
  • merge(p1, p2) — merges two lists
  • backpatch(p, i) — fills target i into all instructions in list p
  • nextquad — the index of the next instruction to be generated

Generated Three-Address Code with Back-Patching

--- OUTER WHILE: while(x < 200 || y < 400) ---
100:  if x < 200 goto 103    ; truelist1 = {100}
101:  goto 102                  ; falselist1 = {101}

Q12

Describe code optimization techniques with examples: Constant folding, propagation, algebraic simplification, strength reduction, CSE

Module 5 15 Marks May 2024
▼

1. Constant Folding

Evaluate constant expressions at compile time rather than at runtime.

// Before:
x = 3 * 4 + 2;
y = 365 * 24 * 60 * 60;   // seconds in a year

// After constant folding:
x = 14;
y = 31536000;

2. Constant Propagation

Replace variables with their known constant values.

// Before:
i = 5;
j = i * 3;
k = j - 1;

// After constant propagation:
i = 5;
j = 15;   // i replaced with 5
k = 14;   // j replaced with 15

// After dead code elimination (i, j never used again):
k = 14;

3. Algebraic Simplification

Apply mathematical identities to simplify expressions.

x = x + 0;    → x = x      (identity for +)
x = x * 1;    → x = x      (identity for *)
x = x * 0;    → x = 0      (zero property)
x = x * 2;    → x = x + x  (if no shift available)
x = a ** 2;   → x = a * a  (power simplification)
y = (x * 4) / 2; → y = x * 2  (algebraic cancel)

4. Operator Strength Reduction

Replace expensive operations with cheaper equivalents.

x * 2   → x << 1    (shift instead of multiply)
x * 4   → x << 2
x / 2   → x >> 1    (shift instead of divide)
x ** 2  → x * x      (power→multiply)

// In loops — the classic example:
// Before:
for(i=0; i<10; i++) {
    a[i] = i * 4;       // multiply each iteration
}

// After strength reduction:
t = 0;
for(i=0; i<10; i++) {
    a[i] = t;
    t = t + 4;          // addition replaces multiplication
}

5. Common Subexpression Elimination (CSE)

If the same expression is computed multiple times and operands haven't changed, compute it once and reuse.

// Before:
t1 = a + b;
t2 = a + b;   // same expression!
x = t1 * t2;

// After CSE:
t1 = a + b;
x = t1 * t1;  // t2 eliminated

// More complex example:
// Before:
x = (a+b)/(b-c) - (a+b)*(b-c);

// Identify CSEs:
t1 = a + b;   // appears twice
t2 = b - c;   // appears twice
x = t1/t2 - t1*t2;
DAG: CSE is naturally represented in a DAG — identical subexpressions map to the same node, so they're computed once.

Q14

Instruction Scheduling, Software Pipelining and impact on compilers for parallel processing

Module 7 10 Marks Nov 2024 (both sets)
▼

Instruction Scheduling

Modern processors can execute multiple independent instructions simultaneously (superscalar, VLIW). The compiler must reorder instructions to maximize parallelism while preserving program semantics.

Key concepts:

  • Dependence graph: Nodes = instructions; edges = data/control/anti dependencies. Scheduling must respect topological order.
  • Data dependence: Instruction B reads a value written by instruction A → A must come before B.
  • Anti-dependence (WAR): B writes a register that A reads → A must come first.
  • Output dependence (WAW): Both A and B write to same register → order must be maintained.
  • Goal: Fill pipeline slots by scheduling independent instructions side-by-side.
// Before scheduling (sequential, wastes pipeline slots):
LOAD  R1, a      ; 3-cycle latency
ADD   R2, R1, b  ; stalls 2 cycles waiting for R1
LOAD  R3, c      ; 3-cycle latency

// After scheduling (interleaved to hide latency):
LOAD  R1, a      ; starts cycle 1
LOAD  R3, c      ; starts cycle 2 (independent of R1)
ADD   R2, R1, b  ; R1 ready by now → no stall

Software Pipelining

Software pipelining overlaps iterations of a loop so that different loop iterations execute simultaneously — essentially creating a "pipeline" of iterations in software.

Technique: Loop pipelining / Modulo scheduling

// Original loop:
for(i=0; i<n; i++) {
    a[i] = b[i] + c[i];   // Load-Add-Store sequence
}
// Each iteration: LOAD b[i], LOAD c[i], ADD, STORE a[i]

// Software pipelined (overlap 3 iterations):
// Prologue: start iterations 1 and 2
LOAD b[0]; LOAD c[0];              ; start iter 0
LOAD b[1]; LOAD c[1]; ADD r0=b0+c0;; start iter 1, compute iter 0
// Steady state: all three stages active
LOAD b[i]; ADD b[i-1]+c[i-1]; STORE a[i-2];
// Epilogue: drain last iterations

Result: Effective throughput approaches 1 iteration per cycle even though each iteration takes 3 cycles.

Impact on Compiler Design

  • Code generation phase must be aware of processor's functional units, latencies, and parallelism capabilities.
  • Register allocation becomes more complex — live ranges extend across iterations in pipelining.
  • Loop transformations (unrolling, loop fission, fusion) are applied to expose more parallelism.
  • Dependence analysis becomes critical — compiler must prove that reordering is safe.
  • Compilers for VLIW/superscalar machines must pack multiple independent operations into a single instruction word.

Automatic Parallelization and Cache Locality

Automatic parallelization: Compiler detects independent loop iterations and generates parallel code (e.g., OpenMP directives, SIMD code). Challenges: aliasing, pointer analysis, complex data structures (graphs, sparse matrices) make dependence analysis undecidable in general.

Cache locality optimization:

  • Temporal locality: Access the same data multiple times before it's evicted. Loop restructuring can improve this.
  • Spatial locality: Access consecutive memory locations. Array traversal order matters (row-major vs column-major).
  • Loop tiling (blocking): Process data in cache-sized blocks to reduce cache misses.
// Poor cache: column-major access in C (row-major language)
for(j=0; j<N; j++)
  for(i=0; i<N; i++)
    A[i][j]++;   // jumps by N elements each access

// Good cache: row-major access
for(i=0; i<N; i++)
  for(j=0; j<N; j++)
    A[i][j]++;   // sequential memory access

Challenges with irregular data structures: Sparse matrices, graphs have unpredictable access patterns → compilers cannot statically prove locality → dynamic scheduling needed → difficult to parallelise automatically.

Identify Common Subexpressions

Expression: x = ((a+b)/(b-c)) - (a+b)*(b-c)

  • (a+b) appears twice → CSE node n1
  • (b-c) appears twice → CSE node n2

DAG Construction

Optimal Target Code from DAG

MOV  R0, a        ; R0 = a
ADD  R0, b        ; R0 = a+b  (n1)
MOV  R1, b
SUB  R1, c        ; R1 = b-c  (n2)
MOV  R2, R0
DIV  R2, R1       ; R2 = (a+b)/(b-c)  (n3)
MUL  R0, R1       ; R0 = (a+b)*(b-c)  (n4) — reuse n1 in R0
SUB  R2, R0       ; R2 = n3-n4  (n5)
MOV  x, R2        ; x = result
